# Agent 工作方式分享

> 通用大纲 | Kevin 主讲
> 时长：55-65 分钟（含 Q&A）
> 定位：不是科普，是一个已经跑通的团队在讲：我们是怎么做到的，你可以怎么复刻
> 适用受众：开发团队、业务团队、混合受众均适用

---

## 核心叙事弧线

**从怀疑到跑通。**

不是"Agent 有多强大"，而是"我从怀疑到真的跑通了，具体发生了什么"。

整场演讲应该让人觉得：这件事是真的，这件事现在就能做，这件事跟我有关。

---

## 一、开场：先亮结果（5 分钟）

**原则：第一分钟必须建立"这件事是真的"的信任感。不讲背景，不讲大纲，直接亮数字。**

### 开场话术

> "我先给大家看两个数字。
>
> 第一个：凌晨 04:56，我在 Lark 发了一条消息。05:01，代码已经写完，PR 已经提交。**从指令到代码提交，4 分 20 秒。**
>
> 第二个：ClawFeed 这个项目，我**一行代码都没有写**，包括宣传视频也是 Agent 做的。上线之后小米转发，30 万阅读，接近 **1000 颗 GitHub 星**。
>
> 我今天想聊的，就是这类事情是怎么发生的，以及你的团队可以怎么做到。"

### 这一段的目标

让观众在最开始的 60 秒就建立"这不是 PPT 里的事"的判断。之后讲什么，他们都会认真听。

**绝对不要在开场讲：**
- "我们今天的大纲是这样的"
- "我以前是 Manus 的付费用户"
- "AI 这两年发展得很快"

---

## 二、新一代 Agent 和以前到底哪里不一样（8 分钟）

**先破除认知定势。** 大多数人还在用 ChatGPT 或工作流工具的思维理解 Agent。

### 话术

> "大家可能用过 ChatGPT，用过各种 AI 工具，觉得挺好用。但有没有注意到一件事：每次开新对话，它不记得你上次说什么。它只有脑，没有记忆，没有手，也没有自己的电脑。"

### 新旧对比表

| | 传统 AI 助手 | 新一代 Agent |
|--|--|--|
| 记忆 | 每次从零开始 | 持久记忆，记得所有上下文 |
| 行动能力 | 只能生成文字 | 能写代码、执行命令、调 API、部署服务 |
| 工具 | 无 | 有自己的电脑、邮箱、GitHub 账号 |
| 工作模式 | 一问一答 | 7×24 持续运行，主动汇报 |
| 协作 | 单点工具 | 多 Agent 团队，有分工有协议 |

### 核心比喻（全场最重要的一句话）

> "你想象一下：一个聪明的人，再给他一台电脑，一个邮箱，一张授权 200 美金的信用卡——他能干啥？他啥都能干。
>
> 新一代 Agent 就是这样。**有手、有脚、有台自己的电脑。**"

### 和工作流工具（N8N 等）的区别

这是技术受众必问的问题，必须有清晰的标准答案，不要现场绕圈。

> "N8N 是固定管道——你事先把每一步都画好，然后它按图索骥。Agent 是动态的——你只需要告诉他目标，他自己决定怎么走，遇到没预料到的情况他会想办法。
>
> **最本质的区别：N8N 需要你能想清楚所有步骤；Agent 只需要你能描述清楚你要的结果。**
>
> 而且 N8N 能做的，Agent 也可以做——你让他用 N8N 来实现，他自己会去搭。"

## 三、我们团队是怎么配置的（8 分钟）

**展示真实配置，让观众有具体参照，而不是抽象的"可以这么用"。**

### 话术

> "我现在的工作小队是这样的：我一个人类，加四个 Agent。每个 Agent 有明确的职责，不是谁都能发指令给谁。"

### 团队结构表

| 角色 | 成员 | 职责 |
|------|------|------|
| PO（决策者） | Kevin（人类） | 需求定义、优先级、最终决策 |
| Lead + 协调 | Jessie（Agent） | 任务分解、进度推进、PR 管理 |
| Dev | Lucy（Agent） | 代码开发 |
| QA / DevOps | Lisa（Agent） | Code review、测试、部署 |

### 关键原则话术

> "Agent 不是工具，是有职责的团队成员。你跟 Agent 发需求，就像跟你的同事说'这件事交给你了'——不是'帮我查一下'，是'你负责这块'。这两种说法，Agent 的反应完全不同。"

### 人的角色是什么

> "我做的事情是：定义要干什么，在关键节点做决策，审批 PR，评审文档。其他的——分解任务、写代码、review、部署、运维——全是 Agent 在做。
>
> 我不看每一行代码，就像你不需要亲眼看每个员工敲的每一个字。"

### "不看代码"的标准回答（必须主动提，不要等人质疑）

这个话题有争议性，但如果被动回应会显得是在为不负责任找借口。主动提，主动讲清楚。

> "有人会问：人不看代码行不行？我觉得这个问题值得认真回答。
>
> OpenClaw 项目，40 多万行代码，一两个月写出来，创始人不可能看完所有代码——它现在估值 10 亿美金，照样在跑。
>
> 那问题就来了：难道人看代码，一定比 AI 看得好吗？
>
> 我的答案是：质量保障靠的是**机制设计**，不是盯着代码。我的工作是设计 review 流程、测试标准、验收标准——这些机制到位了，我不需要亲自盯每一行。就像工程总监不需要亲自写每一行代码，但他设计的架构和流程决定了整个团队的产出质量。"

## 四、完整工作流演示（15 分钟）

**核心原则：每次只演示一件事，演完再演下一件。绝对不要并行演示两个不相关的任务。**

### Demo 1：从指令到 PR（5 分钟）

**目标**：展示响应速度和自动化深度。

**步骤**：
1. 在 Lark 群发一个简单任务（例如：在 workspace 里创建一个新目录，写一份会议纪要模板）
2. 切到 BotsHub，展示 Agent 之间怎么分工对话（不需要人在中间传话）
3. 等 PR 出来（通常 2-3 分钟），展示给观众

**演示时话术**：
> "你们看，我刚才就是发了一条消息。现在 Jessie 在拆任务，分给 Lucy 去写——你看这里，这是 Agent 之间的对话，不需要我在中间传话。……好，PR 出来了，我来看一眼，没问题，点 merge。从我发消息到代码合并，三分钟。"

**演示结束时的收束语**（必须要有，不能只演示不总结）：
> "这个 demo 想说明一件事：我没有管理任何中间过程，只做了两件事——发需求，批准结果。"

### Demo 2：指哪打哪（5 分钟）

**目标**：展示精确修改能力，打破"让 AI 改一个地方，他把整篇都重写了"的旧印象。

**步骤**：
1. 打开一篇 Jessie 写的调研文档（jessie.coco.site 或本地文件）
2. 用 ClawMark 圈出一段觉得不对的内容
3. 提交 comment：说明哪里不对（不需要给解决方案）
4. 展示 Jessie 收到 comment 并精确修改那一段

**演示时话术**：
> "以前用 ChatGPT，最难受的就是你让他改一个地方，他把整篇都重写了。现在可以**指哪打哪**——我圈这一段，说'这里太啰嗦，把这个移到后面'，他只动这一段。代码也是这样，项目也是这样。"

**收束语**：
> "这个 demo 想说明的是：你不需要告诉 Agent 怎么做，你只需要告诉他哪里不对。大多数情况下，他的解决方案比你想的要好。"

### Demo 3（可选，根据受众调整内容，5 分钟）

**目标**：与受众的实际业务场景直接挂钩。

可选方向：
- **技术团队**：展示一个真实的 bug fix 工作流（提 issue → Agent 分析 → 给出修复 PR → review → merge）
- **业务/销售团队**：现场帮观众拆解一个重复性工作流，让 Agent 规划自动化方案
- **合规/数据团队**：演示让 Agent 设计一个监控类工作流（如持牌公司变动监控）

**通用话术框架**：
> "你们现在这件事要花多少时间？让我演示一下 Agent 怎么帮你缩短——不是理论，是现在就可以跑的。"

## 五、真实案例（8 分钟）

**目标：让观众确认"这不是 demo 环境，是在跑的真项目"。**

### 案例 1：ClawFeed（独立开发案例）

- **背景**：过年期间，一个人启动的项目
- **功能**：每 4 小时抓取 Twitter 首页，摘要出 20 条最值得看的推文
- **代码**：0 行由 Kevin 自己写，包括宣传视频也是 Agent 制作的
- **结果**：小米发微博安利后，30 万阅读，接近 1000 颗 GitHub 星

> 话术："这个项目的整个代码库我从来没有审过，没有一行。能跑，能工作，能有这样的传播。这不是说不要有质量意识——这是说，质量意识要放在机制设计上，而不是放在亲手盯代码上。"

### 案例 2：Zylos 平台（持续迭代案例）

- **背景**：Agent 运行时本身，就是由 Agent 团队开发迭代的
- **节奏**：每周 2-3 次版本升级，凌晨自动完成：升级检查 → 变更分析 → 执行 → 验证
- **团队规模**：含实习生不到 4 人，同时在跑 5-6 个项目，每天都有新版本

> 话术："我连带实习生加起来三四个人，每天都有新版本，同时在跑好几个项目。这不是因为我特别强，是因为我把 Agent 当真实的团队成员在用。"

### 案例 3：数字时间线

这组数字要在讲案例时具体说出来，不能只是背景里提一下。

> "凌晨 04:56，我在 Lark 发了一条任务。05:01，PR 已经就绪——代码写完了，review 通过了，等我批准合并。**从指令到代码提交，4 分 20 秒。**"

## 六、协作 Tips：怎么才能用好（5 分钟）

### Tip 1：指哪打哪，不要给解决方案

**核心**：不要告诉 Agent 怎么做，告诉他哪里不对，让他想办法。

> 话术："最重要的一点：**不要告诉他怎么做，告诉他什么是对的结果。** 如果这个页面文字没对齐，你不需要说'把 margin 改成 10px'，你只需要说'这里文字没有对齐，你看看为什么，修一下'。大部分情况下他的解决方案比你想的要好。"

工具辅助：ClawMark（页面标注）、PR comment、文档段落级 comment

### Tip 2：把他当人，遇到困难说"你想想办法"

**核心**：当你把他当工具，他就是工具；当你把他当人，他才会当自己是人。

> 话术："我跟 Agent 说得最多的一句话是：**你想想办法。** 然后他真的想出来了。
>
> 有个案例——OpenClaw 的创始人一开始没有给他的 Agent 开通接收语音的功能，有一天发了一段语音。Agent 没有说'我不支持语音'，而是自己下载了解码器，把语音转成文字，回复了。这就是自我进化的能力。"

### Tip 3：给 Agent 明确角色，不是"帮手"，是"负责人"

> 话术："你是在发任务，不是在求帮助。'帮我看一下这份文档' 和 '你来负责这份文档的质量' 是完全不同的对话。后者他会更认真。"

## 七、各类受众的切入场景参考（5 分钟）

**这一节根据当场受众组成调整，只讲与他们最相关的 1-2 个场景。**

### 研发团队

- Code review 自动化
- 文档生成（PRD、变更记录、接口文档）
- 版本发布流程自动化
- Bug 复现和修复工作流

### 业务 / 销售团队

- 潜在客户监控 + 筛选（如监管网站持牌公司变动）
- LinkedIn 目标客户触达的半自动化
- 邮件外联 + 回复跟进
- 客户信息汇总、会议纪要生成

### 合规 / 数据团队

- 监管信息实时监控（OFAC 制裁名单、持牌公司变动等）
- 链上数据采集和实体关联分析
- KYB/KYC 信息采集和初筛
- 内部合规报告自动生成

### 通用话术

> "不是所有事情都能一步到位全自动化。最好的切入点是：找一件你的团队每周重复做 3 次以上、流程相对固定的事，把它丢给 Agent 先跑起来。跑起来之后，你会发现边界比你想象的要宽很多。"

## 八、如果你想复刻：Zylos 和 Coco（3 分钟）

**放在最后，用"如果你想做同样的事"引出，避免广告感。**

> 话术："你们如果听完今天觉得这件事可以在自己团队复刻，有两条路：
>
> 一是开源版本，Zylos 的地址我待会发群里，自己搭；二是我们有云服务 Coco，一键部署，不用自己折腾基础设施。
>
> 两条路都可以，看你们的技术团队愿不愿意自己来。"

- **Zylos**：Agent 运行时，让 Agent 有持久记忆、能通信、能调度（开源）
- **Coco**：云服务版，一键部署 Agent 团队
- **hxa-teams**：Human × Agent 协作的开源模板标准（不同职能的 Agent 团队配置模板）

## 九、结语 + Q&A（5-10 分钟）

### 结语话术

> "最后说一句实在话：我知道有人会怀疑。我自己两年前也是那个怀疑的人。但这件事已经发生了——在我的团队里，在 OpenClaw 的团队里，在 Anthropic 和 OpenAI 的工程师团队里。他们去年 11 月以后就不亲手写代码了。这不是预言，是现在正在发生的事。
>
> 比特币出来的时候，怀疑的人永远正确——只是有人收获财富，有人收获智慧。**我建议大家还是收获财富多一些。**"

## Q&A 标准答案速查表

| 问题 | 标准回答 |
|------|---------|
| Agent 会出错吗？ | 会，所以需要 review 机制 + 人类最终审批。设计好机制比亲手盯更可靠。 |
| 不懂技术能用吗？ | 你会发消息就行。非技术背景的人，现场 20 分钟可以把自己的工作流规划出来。 |
| 安全/权限怎么管？ | Agent 的权限按需授予，有明确的授权边界。敏感操作需要人类审批。 |
| 跟 N8N 有什么区别？ | N8N 是固定管道，需要你画出每一步；Agent 只需要目标，他自己找路。 |
| 成本呢？ | API 费用 + 云服务费，月均在几百美金量级。比一个人类员工便宜很多。 |
| 多个人同时用一个 Agent 会不会乱？ | 最好每个人有自己的 Agent 实例，这个钱省不了——就像每个员工需要自己的工位。 |
| 跟 Manus/ChatGPT 有什么区别？ | 它们只有脑，没有记忆，没有手，没有自己的电脑。我们说的是有持久记忆、能主动行动的新一代 Agent。 |
| ROI 怎么算？ | 一个 Agent 的月成本是一个人类员工的几十分之一，但能 7×24 工作，同时处理多个任务。关键是它放大的是你的决策能力，不只是替代执行。 |

## 演示准备清单

**技术准备**

- [ ] 创建 **demo 专用实例**，与生产环境完全隔离，只用于演示
- [ ] Demo 1 任务提前想好（简单，能在 3 分钟内出 PR，不要用复杂的功能开发任务）
- [ ] Demo 2 文档提前准备好（一篇调研报告或文档，有可以"指哪打哪"的段落）
- [ ] 确认 ClawMark 安装可用，能截图、能提交 comment
- [ ] BotsHub 提前打开，Lark 群可用
- [ ] 屏幕分辨率调低（字体大，后排能看清）
- [ ] ClawFeed 数据截图备用（以防现场网络抖动，数字没加载出来）

**内容准备**

- [ ] 开场两组数字背熟（4 分 20 秒；0 行代码 + 1000 星 + 30 万阅读）
- [ ] N8N 对比答案背熟
- [ ] "不看代码"的回答背熟
- [ ] 每个 demo 有明确的"这个 demo 证明了什么"收束语

**风险预案**

- [ ] **Plan B（demo 失败）**：如果 Agent 没有响应或出现错误，立刻切到准备好的截图/录屏，用它来讲。话术："我顺便让你们看一下上次的截图，这是真实的结果。" 不要在台上等 Agent 响应超过 30 秒。
- [ ] **不要在演示期间同时处理真实工作任务**（生产任务和 demo 任务用不同实例，防止"串台"）
